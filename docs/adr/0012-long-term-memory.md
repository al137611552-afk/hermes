# ADR-0012: 长期记忆（跨会话）

- 状态：已接受
- 日期：2026-06-09

## 背景
P6.1 持久化让单个会话可重开恢复，P6.2 压缩控制单会话上下文体积——但二者都局限在
「一个会话内」。用户跨会话反复出现的事实（称呼/偏好、项目目标与约束、达成的关键决定）
在每个新会话里都要重讲一遍，模型也「记不住」。PRD FR-6.3 要求长期记忆：跨会话、跨重启
持久，且能在新会话里自动被模型用上。参考形态即 Claude Code 自身的 file-based 记忆机制。

## 决策
- **独立存储，与会话库解耦**：记忆是全局的（不属于任何单一会话），放独立 SQLite 文件
  `data/memory.db`（`MemoryStore`，自带连接 + 锁），不与 `hermes.db` 的 sessions/messages
  混表，也不受会话删除/GC 影响。无新依赖（标准库 sqlite3）。
- **记忆是「短事实」条目**：`{id, content, kind, source, created_at, updated_at}`，
  `kind ∈ {user, preference, project, fact}`（非法值归一为 fact）。按 content 折叠空白后
  去重，避免反复记同一条。
- **三种写入/使用路径并存**：
  1. **模型工具** `remember / recall / forget`（非危险，不过权限 gate，作为工具块在 UI 可见）：
     模型在对话中可主动记、查、删。
  2. **自动抽取**：离开会话时（`new_session` / 切到别的 `load_session`）后台线程把刚结束的
     对话喂给模型，要求输出 JSON 记忆条目，去重后入库。只抽取「自上次抽取以来的新消息」，
     用 `_extracted_upto[session_id]` 占位防并发重复；失败静默（best-effort）。
  3. **回忆注入**：每次 `send_message` 前，把记忆（按条数 + 字符预算，最近优先）拼成
     `[长期记忆]` 块追加进 system，再走 P6.2 预算压缩。这样**全新会话也「记得」**。
- **纯逻辑与存储分离**：`store/memory.py` 只管 SQLite；`longmem.py` 放可单测的纯函数
  （注入块拼装、对话转录、抽取提示构造、模型输出解析），不依赖网络。
- **开关**：`memory.enabled`（总开关，关掉则不注册工具、不注入、不抽取）、
  `memory.auto_capture`（仅关自动抽取，保留工具）、`max_inject` / `max_inject_chars`
  （注入预算）、`min_messages_to_capture`（触发抽取的最小消息数）。

## 备选与权衡
- **每条消息后就抽取**：实时但成本高（每轮多一次模型调用）、且对话未结束时要点未定。
  否决——改为「离开会话」这一自然边界触发。
- **存进 hermes.db 同库加表**：少一个文件，但与会话生命周期/外键/GC 耦合，且记忆本是全局的。
  否决——独立库语义更清晰。
- **向量检索做相关性召回**：更精准但需嵌入模型/向量库依赖。当前个人本地、记忆条数有限，
  「全量按预算注入 + 工具关键词 recall」已够；保留后续升级空间（条数大了再上检索）。
- **file-based（对标 Claude Code）**：可读可手改，但与本项目既有 SQLite 持久化不一致、
  且要自管索引。否决——复用 SQLite 更省。

## 已知限制
- 自动抽取依赖主模型听话输出 JSON；解析容错（去围栏 / 截取 `{...}`）已做，仍可能偶尔抽空。
- 抽取占位 `_extracted_upto` 在内存里，进程重启清零——重启后再离开同一会话可能重抽一次，
  但 `add` 去重兜底，最多生成等价条目被跳过。
- 注入是「全量按预算」，记忆很多时只截断到预算内（最近优先），非语义相关性排序。
- 无 GUI 管理面板（本期范围外）：查看/编辑/删除靠模型 `recall`/`forget` 工具；后续可加面板。
- 删除一个会话不会回收其自动抽取出的记忆（记忆已是独立事实，刻意保留）。

## 结果
- 跨会话、跨重启的事实得以沉淀并在新会话自动生效，减少重复交代。
- 工具 + 自动抽取 + 注入三路径互补；纯逻辑可单测（test_p6_memory 8/8）。
- 与 P6.1/P6.2 正交：DB 完整历史、单会话压缩均不受影响。

# ADR-0009: 上下文 token 预算与压缩

- 状态：已接受
- 日期：2026-06-09

## 背景
P6.1 持久化后，会话历史会跨重启不断累积。每次 `send_message` 把整段 `Api.history`
喂给模型，长会话会撑爆模型上下文窗口、并线性抬高 token 成本。
P6.2（PRD FR-6.2）要求在喂给模型前按 token 预算裁剪上下文。

## 决策
- **启发式 token 估算，不引分词器依赖**（延续 P6.1「标准库、无新依赖」）：
  ASCII 约 4 字符/token、非 ASCII（中文等）约 1 字符/token，图片块按固定量级估值。
  仅用于预算决策，宁可略高估，不追求与某家分词器精确一致。
- **保留近窗 + 旧段摘要**：超预算时从最早的回合整段丢弃，直到剩余 ≤ 预算；
  被丢弃内容压成一段文本摘要追加进 system。至少保留最近 `keep_recent_turns` 个回合。
- **裁剪点只落在「真实用户回合」边界**：一次 tool-use 往返里，assistant 的 `tool_use`
  必须与其后 user 的 `tool_result` 成对，否则 Anthropic API 报错。因此切点只选
  `role==user 且不含 tool_result` 的消息处，绝不从工具往返中间切断。
- **只裁喂给模型的副本，不动持久化**：DB / `Api.history` 始终是完整历史；压缩作用于
  本次请求的副本。写回时按「压缩后喂入条数」切出本轮新增的 assistant/tool 消息，
  extend 回完整 `history` 并落库——避免压缩污染历史或错位持久化索引。
- 由 `config.context`（`enabled` / `max_input_tokens` / `keep_recent_turns`）控制，可关闭。
- 触发压缩时 emit `context_compressed` 事件，前端显示一条提示（历史仍完整保存）。

## 备选与权衡
- **用模型对旧段做语义摘要**：质量更高，但多一次 API 调用（慢、花钱）、且摘要本身要再纳入
  预算 —— 第一版否决，采用零调用的截断式摘要；语义摘要留作后续可选增强。
- **精确分词（tiktoken / API count_tokens）**：tiktoken 增依赖且与非 OpenAI 模型不符；
  API 计数要联网。启发式估算对「是否超预算」的判断已足够 —— 采用启发式。
- **每轮 loop 内都压缩**：更彻底（工具往返也会涨），但增 loop 复杂度。第一版只在
  `send_message` 进 loop 前压一次，覆盖跨会话累积这一主要膨胀源；单轮内增长留作已知点。

## 已知限制
- 估算是粗略的，极端构成（大量代码/表情符号）可能偏差较大；预算建议留足余量。
- `keep_recent_turns` 是下限保护：当最近若干回合本身已超预算时，不再继续切（避免破坏
  tool 配对或丢光上下文），此时实际输入可能仍超预算，交由模型/上层处理。
- 单个超大回合（无可切边界）不裁剪，原样发出。

## 结果
- 长会话喂模型前自动裁剪到预算内，控制上下文溢出与成本，且不破坏 tool-use 往返。
- 持久化历史不受影响，用户在 UI 上仍看到完整对话。
- 为 P6.3 长期记忆（被裁内容可改为进记忆而非纯丢弃）预留了接入点。
